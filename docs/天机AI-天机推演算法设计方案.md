# 天机AI商业决策智能体：天机推演算法设计方案

> 版本：v0.1  
> 日期：2026-06-11  
> 定位：将「企业问题入口 → 数据/资料沉淀 → 知识图谱/RAG → 多角色决策推演 → 风险审计 → 验证计划 → 经营档案 → 方法论复利」落成一套可执行的商业决策推演算法。

---

## 1. 设计结论

天机AI不应只是一个知识库问答系统，也不应只是一个商业画布报告生成器。

它真正要解决的问题是：

> 当用户提出一个企业诉求时，系统要帮助用户看见这件事未来可能走向哪里、哪些变量会改变走向、哪些假设必须验证、哪些风险会影响决策结果，并最终形成可执行的验证计划。

因此，天机的核心算法应定义为：

## 天机多路径商业推演算法

英文暂定：

**Tianji Multi-Path Business Simulation Algorithm**

简称：

**Tianji-MPS**

它的核心不是「回答一个问题」，而是「生成一张决策岔路图」。

---

## 2. 从 MiroFish 借鉴什么

MiroFish 的核心价值不在于视觉形态或鱼群概念，而在于它形成了一条完整的推演链路：

1. 上传现实材料作为种子。
2. 从材料中抽取知识图谱。
3. 基于图谱生成具备角色、记忆、行为逻辑的智能体。
4. 让智能体在模拟环境中交互。
5. 汇总交互结果，生成预测报告。

对应到 MiroFish 的系统结构，可以概括为：

```text
Seed Materials
→ Ontology / Knowledge Graph
→ Persona / Agent Profiles
→ Simulation Environment
→ Multi-Agent Interaction
→ Report Agent
```

这个结构对天机有非常强的启发意义。

但天机不能照搬 MiroFish，原因有三点：

1. MiroFish 更偏舆论、社会反应、群体行为预测；天机要解决的是企业经营决策。
2. MiroFish 的 Agent 更像模拟人群；天机需要模拟商业决策中的利益相关方。
3. MiroFish 的输出是预测报告；天机的输出必须进一步形成验证任务、经营档案和方法论复利。

所以，天机要吸收的是「图谱驱动的多智能体推演」，而不是「社交平台舆论仿真」。

---

## 3. 天机算法的目标

天机推演算法要回答的不是：

> 这个项目行不行？

而是：

> 如果这个项目继续推进，最可能出现哪些路径？每条路径的关键变量、风险、证据缺口和验证动作是什么？

算法输出应具备五个特征：

1. **多路径**：不只给一个答案，而是给出多种可能走向。
2. **有约束**：所有推演必须基于用户资源、时间、成本、渠道、客户和风险边界。
3. **有角色**：从不同利益相关方视角进行判断。
4. **有证据**：引用知识节点、历史案例、资料内容、对话沉淀和用户输入。
5. **可执行**：最终必须落到验证卡、行动计划和经营档案。

## 4. 核心算法链路

```text
用户企业诉求
→ 问题框定
→ 证据召回
→ 决策角色生成
→ 多路径推演
→ 交叉因果链分析
→ 风险审计
→ 验证计划生成
→ 经营档案沉淀
→ 知识节点候选更新
```

## 5. 模块一：企业诉求解析

### 5.1 输入

用户可能输入的是非常模糊的问题，例如：

- 我想做一个面向老人的智能药盒，是否可行？
- 我想在成都做 OPC 社群，这个模式有没有机会？
- 我们公司渠道成本太高，价值主张是不是出了问题？
- 帮我判断这个合作合同是否值得签。

系统不能直接回答，而要先拆解成结构化问题。

### 5.2 解析字段

```text
decision_objective       决策目标
business_context         企业/项目背景
target_customer          目标客户
current_problem          当前核心问题
available_resources      当前资源
constraints              约束条件
stakeholders             利益相关方
time_horizon             时间窗口
risk_tolerance           风险偏好
unknown_assumptions      未验证假设
expected_output          用户期待输出
```

### 5.3 输出

系统应生成一份「问题框定卡」：

```text
用户真正要判断的不是 A，而是 B。
当前最关键的不确定性是 C。
如果 C 没有被验证，后续报告、计划和判断都可能失真。
```

---

## 6. 模块二：证据图谱召回

### 6.1 召回对象

天机不应只做向量相似度召回，而应结合图谱关系召回。

召回对象包括：

1. 港大 IMC&IPM 核心方法论节点。
2. 老师课件形成的知识节点。
3. 已审核企业资料。
4. 用户上传资料。
5. 历史对话结果。
6. 历史诊断报告。
7. 验证卡与执行反馈。
8. 失败假设和风险案例。

### 6.2 知识分层

```text
L0 港大 IMC&IPM 核心方法论
L1 老师课件与课程知识节点
L2 已审核案例与同学经验
L3 用户企业资料
L4 用户对话结果与验证反馈
```

### 6.3 召回原则

1. L0/L1 是主干，不可被 L3/L4 覆盖。
2. L3/L4 用于补充场景和事实，不直接替代方法论判断。
3. 每次回答必须记录「引用了哪些知识节点」。
4. 高价值对话结果不能自动污染主干知识库，必须作为候选资产进入审核。

## 7. 模块三：决策角色生成

MiroFish 生成的是社交媒体用户或事件相关人物。

天机生成的是商业决策中的角色。

### 7.1 标准角色池

```text
CEO / 创始人
CFO / 财务负责人
目标客户
渠道伙伴
竞争对手
运营负责人
增长负责人
法务与合规
产品负责人
港大方法论审计员
```

### 7.2 每个角色的结构

```text
role_name             角色名称
decision_goal         该角色最关心什么
risk_preference       风险偏好
evidence_focus        优先看哪些证据
opposition_style      主要反对角度
success_metric        成功判断指标
blind_spot            可能忽略的问题
```

### 7.3 示例

如果用户提出「做银发族智能药盒」，系统应生成：

- 创始人：关注市场机会、增长速度、产品差异化。
- CFO：关注 BOM 成本、获客成本、订阅收入是否覆盖成本。
- 老人用户：关注是否真的容易使用、是否产生负担。
- 子女用户：关注是否愿意付费、是否能降低照护焦虑。
- 药房渠道：关注是否能带来复购、推荐是否有收益。
- 合规角色：关注医疗器械、数据隐私、健康建议边界。
- 竞争对手：评估大厂复制难度和渠道壁垒。
- 方法论审计员：检查价值主张、客户细分、渠道通路是否自洽。

## 8. 模块四：多路径推演

### 8.1 为什么要多路径

企业问题很少只有一种结果。

同一个项目可能因为客户反应、渠道成本、竞争对手、监管政策、现金流、团队能力不同，走向完全不同。

所以天机要输出的不是单一判断，而是多条路径。

### 8.2 标准路径类型

```text
路径 A：保守验证路径
路径 B：进攻增长路径
路径 C：低成本试错路径
路径 D：资源不足失败路径
路径 E：竞争对手反制路径
路径 F：合规/现金流风险路径
```

### 8.3 每条路径的字段

```text
path_name               路径名称
trigger_conditions      触发条件
key_assumptions         核心假设
expected_outcome        可能结果
leading_indicators      早期信号
risk_points             风险点
recommended_action      建议动作
stop_loss_condition     止损条件
next_validation         下一步验证
```

### 8.4 示例

```text
路径 A：先验证子女付费意愿
触发条件：老人使用意愿不确定，子女是潜在付费方。
核心假设：子女愿意为降低照护焦虑付费。
早期信号：访谈中子女愿意留下联系方式，愿意试用原型。
止损条件：连续 20 个目标用户中，低于 3 人愿意付费。
下一步验证：制作非硬件原型，测试安心报告是否有价值。
```

---

## 9. 模块五：交叉因果链

多路径推演之后，系统要进一步分析变量之间的因果关系。

### 9.1 因果链结构

```text
用户需求变化
→ 价值主张调整
→ 渠道通路变化
→ 获客成本变化
→ 收入来源变化
→ 成本结构变化
→ 现金流压力变化
→ 项目成败概率变化
```

### 9.2 典型因果链

```text
如果目标客户从老人切换为子女
→ 价值主张从“健康提醒”变成“降低照护焦虑”
→ 渠道从药房转向社区/社群/家庭场景
→ 产品不一定先做硬件
→ 可先验证订阅服务
→ 成本压力下降
→ MVP 成功概率上升
```

### 9.3 输出要求

每次推演至少输出 2-3 条因果链：

1. 成功因果链。
2. 失败因果链。
3. 反直觉因果链。

## 10. 模块六：风险审计

### 10.1 风险分类

```text
客户风险
需求风险
渠道风险
成本风险
现金流风险
竞争风险
合规风险
数据风险
执行风险
组织能力风险
```

### 10.2 风险矩阵字段

```text
risk_name          风险名称
risk_type          风险类型
probability        发生概率
impact             影响程度
reversibility      是否可逆
evidence_level     证据强度
mitigation         缓解动作
owner              责任角色
deadline           处理时间
```

### 10.3 审计原则

1. 不允许只输出「有风险」，必须说明风险如何发生。
2. 不允许只输出「建议关注」，必须给出验证动作。
3. 对不可逆风险，优先级高于增长机会。
4. 对没有证据的高影响风险，必须进入验证计划。

---

## 11. 模块七：验证计划生成

天机的推演不能停在分析层，必须进入验证层。

### 11.1 验证计划类型

```text
7 天验证卡
30 天验证计划
90 天行动计划
MVP 最小验证路径
关键假设验证清单
```

### 11.2 验证卡字段

```text
validation_title       验证任务标题
hypothesis             要验证的假设
why_now                为什么现在要验证
target_user            验证对象
method                 验证方法
success_metric         成功指标
failure_signal         失败信号
minimum_cost           最低成本做法
timeline               时间安排
next_decision          验证后如何决策
```

### 11.3 验证计划原则

1. 优先验证最危险的假设，而不是最容易做的动作。
2. 优先用低成本方法验证，不急于开发完整产品。
3. 每个验证动作必须绑定成功指标和失败信号。
4. 验证结果必须回流经营档案。

---

## 12. 模块八：经营档案沉淀

每次推演结果都要沉淀到 Project。

### 12.1 沉淀对象

```text
用户原始诉求
问题框定结果
召回知识节点
多角色推演过程
路径推演结果
风险审计结果
验证计划
用户后续反馈
最终执行结果
```

### 12.2 沉淀价值

经营档案不是资料夹，而是企业决策记忆。

它要回答：

1. 这个项目过去问过什么？
2. 系统曾经判断过什么？
3. 哪些假设被验证了？
4. 哪些判断后来被证明是错的？
5. 哪些方法论在这个企业场景中反复有效？

---

## 13. 模块九：方法论复利

天机最长期的价值不是单次回答，而是方法论复利。

### 13.1 复利来源

```text
老师课件
核心知识节点
企业真实问题
AI 诊断报告
用户执行反馈
失败案例
人工审核结论
```

### 13.2 更新规则

1. 核心方法论只读，不被普通用户数据覆盖。
2. 用户对话结果先进入候选池。
3. 通过人工审核后，才能成为案例节点或扩展节点。
4. 失败案例与错误假设也要沉淀，因为它们能提升后续风险识别能力。

## 15. Prompt 编排建议

### 15.1 第一阶段：问题框定 Prompt

目标：不急于回答，先判断用户真正要决策的问题。

输出：

```text
你真正要判断的是：
当前最关键的不确定性是：
我需要优先验证的假设是：
```

### 15.2 第二阶段：证据召回 Prompt

目标：基于用户问题召回知识节点、案例、历史对话和验证卡。

输出：

```text
本次推演引用的核心方法论：
本次推演引用的案例：
本次推演缺失的证据：
```

### 15.3 第三阶段：多角色推演 Prompt

目标：让多个商业角色分别判断，而不是让一个模型给单一答案。

每个角色必须输出：

```text
我支持什么：
我反对什么：
我最担心什么：
我要求先验证什么：
```

### 15.4 第四阶段：路径归纳 Prompt

目标：将多角色意见归纳成多条走向。

输出：

```text
路径一：
路径二：
路径三：
最危险的路径：
最值得低成本验证的路径：
```

### 15.5 第五阶段：行动计划 Prompt

目标：生成验证卡、行动计划和止损条件。

输出：

```text
7 天内先做什么：
30 天内判断什么：
90 天内是否扩大投入：
什么情况下必须停止：
```

## 16. 产品交互落地

### 16.1 AI 经营访谈

AI 经营访谈是天机推演算法的主入口。

用户输入诉求后，系统应：

1. 自动建立或选择 Project。
2. 生成问题框定卡。
3. 召回相关知识节点。
4. 输出初步多路径判断。
5. 给出 4 个建议追问。
6. 支持一键进入项目验证诊断。

### 16.2 项目验证诊断

项目验证诊断是天机推演算法的深度模式。

它不只是填写九宫格，而是：

```text
九宫格画布
→ 多角色推演
→ 风险审计
→ 单位经济测算
→ MVP 验证路径
→ 90 天行动计划
```

### 16.3 经营档案

经营档案是算法结果的长期承载层。

它要聚合：

```text
对话
资料
报告
验证卡
复盘
知识节点候选
```

## 18. MVP 实施阶段

### 阶段 1：Prompt 级多角色推演

不新增复杂仿真引擎，先用 DeepSeek V4 + 结构化 Prompt 实现。

交付：

1. 问题框定。
2. 5 个固定角色推演。
3. 3 条路径输出。
4. 风险审计。
5. 验证计划。

### 阶段 2：图谱增强推演

引入知识图谱关系，不只用向量召回。

交付：

1. 按节点关系扩展召回。
2. 方法论节点优先。
3. 案例节点辅助。
4. 对话结果候选沉淀。

### 阶段 3：经营档案驱动推演

让系统读取用户项目历史。

交付：

1. 同一 Project 下读取历史判断。
2. 识别前后矛盾。
3. 追踪假设验证状态。
4. 自动更新项目风险画像。

### 阶段 4：轻量仿真引擎

借鉴 MiroFish 的 Agent Simulation，但不做舆论平台模拟。

交付：

1. 多角色多轮辩论。
2. 每轮更新立场。
3. 记录角色冲突。
4. 输出共识、分歧和最小验证动作。

## 20. 关键风险

### 20.1 推演变成表演

如果角色只是换个口吻说同样的话，多角色推演没有价值。

应对：

每个角色必须有独立目标函数、反对理由和验证要求。

### 20.2 输出太长，用户无法行动

如果系统生成长篇报告，用户会觉得专业但不行动。

应对：

每次输出必须收敛到 1-3 个关键验证动作。

### 20.3 知识沉淀污染

用户对话结果有价值，但不能直接进入核心知识库。

应对：

对话沉淀进入候选池，必须通过人工审核。

### 20.4 算法过度自信

商业决策没有绝对正确答案。

应对：

所有判断必须标注假设、证据强度和止损条件。

## 22. 一句话总结

天机推演算法的核心不是让 AI 代替企业做决策，而是让企业在行动前提前看见：

```text
可能路径
关键变量
利益冲突
风险位置
验证动作
止损条件
长期沉淀
```

这才是天机AI区别于普通大模型问答和普通咨询报告的关键。
