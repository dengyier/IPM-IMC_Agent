# 天机AI 落地路线图：从「画布诊断工具」到「30天项目验证系统」

> 原则：不推倒重做，在现有底座上加「任务化 / 闭环化」一层。先定地基（最贵改），再按「降门槛入口 → 任务化输出 → 复盘闭环」分阶段交付。桌面端不破坏，移动端已完成首页/对话/诊断/我的四页骨架。

## 现状（as-is）一句话
强在「方法论严谨 + 画布诊断」，弱在「用户拿到报告就断了」。`DiagnosisReport` 已含 `core_tensions / risk_matrix / mvp_validation_path / ninety_day_plan / final_recommendation / key_assumptions`，即**「五块结构 + 7天/30天计划」数据层已基本就位**，缺的是「项目聚合 + 执行态 + 复盘闭环」。

## 阶段总览

| 阶段 | 目标 | 关键产出 | 依赖 |
|---|---|---|---|
| **0 地基** | 引入「项目」实体 + 诊断内核扩展 | `Project` 模型、`DiagnosisReport.project_id`、报告状态机、`task_pack` 参数 | 无（最先做） |
| **1 入口** | 降门槛 | 首页「你想验证什么」四入口、3分钟体检（一段话→六点） | 0 |
| **2 任务化** | 诊断→可执行任务（闭环左半） | `ValidationTask` 模型、7天/30天可勾选任务卡、报告中心升级为「决策档案」 | 0 |
| **3 访谈式** | 被动回答→主动追问 | 任务包槽位模板、追问引擎、反向验证模式 | 1 |
| **4 复盘闭环** | 回填→复盘→继续/调整/暂停（闭环右半） | 复盘回填接口、复盘结论、我的页升级为「经营档案」、提醒 | 2 |
| **5 底座+多角色** | 资料调用 + 园区/顾问版 | 资料中心沉淀为上下文、知识节点调用型、批量诊断（独立大版本） | 2/4 |

## 执行顺序与 MVP 边界
- 必须按序：**0 → 1 → 2**（0 是地基，2 依赖 0 的 Project/状态机）。
- 推荐 **MVP = 0 + 1 + 2**：诊断→任务，不依赖用户回填即有完整价值。
- 阶段 3、4 作第二迭代，可并行或择一先做。
- **园区版 / 顾问版属不同产品形态（B2B2C），独立大版本，不进 MVP。**

## 最该先拍板
1. MVP 是否锁定「单用户·单项目·诊断→任务包」；园区/顾问延后。
2. 阶段 0 现在就建 `Project`（强烈建议：是，越早越省）。
3. 闭环切两段：先左半（任务化），回填/复盘作第二段。

## 主要风险
- 追问质量（阶段3，需评测集）；
- 30天回填留存（阶段4，需提醒 + 极简选项化回填）；
- 报告状态机复杂度（阶段2，提前把状态流转图定死）。
